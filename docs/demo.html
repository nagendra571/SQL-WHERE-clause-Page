SQL WHERE clauses – CASE vs Boolean

🔧 The “CASE” trap

WHERE EmpID = CASE WHEN @ReturnAll<>1 THEN @EmpID ELSE EmpID END

❌ Forces a table scan – the engine must evaluate the CASE for every row.

✅ Boolean rewrite

WHERE (@ReturnAll = 1) OR (EmpID = @EmpID)

✔️ The optimizer can short‑circuit; if @ReturnAll = 1 it never looks at EmpID.

🧠 Logical equivalence

“If A then B” ⇔ “Not A OR B”.

📚 Real‑world examples

Example 1 – Return all or a single employee

-- CASE version (slow)
WHERE EmpID = CASE WHEN @ReturnAll<>1 THEN @EmpID ELSE EmpID END
-- Boolean version (fast)
WHERE (@ReturnAll = 1) OR (EmpID = @EmpID)

Example 2 – Multi‑column optional filter

-- CASE version (3 CASEs)
WHERE Date = CASE WHEN @Filter=1 THEN @Date ELSE Date END
AND Cust = CASE WHEN @Filter=1 THEN @Cust ELSE Cust END
AND Emp = CASE WHEN @Filter=1 THEN @Emp ELSE Emp END
-- Boolean version (single OR)
WHERE @Filter <> 1 OR (Date = @Date AND Cust = @Cust AND Emp = @Emp)

Example 3 – Optional parameter (NULL = ignore filter)

-- ISNULL/COALESCE (bad)
WHERE CustID = ISNULL(@CustID, CustID)
-- Boolean (good)
WHERE (@CustID IS NULL) OR (CustID = @CustID)

🗒️ Cheat‑sheet (copy‑paste)

-- IF A THEN B
WHERE (NOT A) OR B

-- IF A THEN B ELSE C
WHERE ((NOT A) OR B) AND (A OR C)

-- Optional filter (parameter may be NULL)
WHERE (@Param IS NULL) OR (Column = @Param)
