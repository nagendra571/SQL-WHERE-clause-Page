SQL WHERE clause: Ditch CASE, embrace Boolean logic

When you need to filter rows based on optional parameters, the first instinct is often to reach for a CASE expression inside the WHERE clause. It works, but it pays a hidden price in readability, portability and – most importantly – performance.

In this post I’ll show you how to replace those CASE‑laden filters with plain AND / OR / NOT logic. The result is:

  • 📦 Portable – works on every major RDBMS (SQL Server, Oracle, PostgreSQL, MySQL …).
  • 👀 Easy to read – a developer can understand the intent in a single glance.
  • ⚡ Fast – the query optimizer can use indexes and short‑circuit evaluation.

1️⃣ The logical secret: “IF A THEN B” = “NOT A OR B”

In propositional logic an implication p → q is equivalent to ¬p ∨ q. That is, “if A then B” is true whenever A is false **or** B is true.

Both columns “A → B” and “¬A ∨ B” have identical truth values, which proves the equivalence. (See Wikipedia for the formal proof – p → q ≡ ¬p ∨ q)【13†L60-L66】.

Why this matters for SQL

If you turn an “IF A THEN B” test into “NOT A OR B”, the optimizer sees a plain Boolean expression. It can then:
  • use index seeks on the B part only when A is false,
  • short‑circuit the evaluation (skip B entirely when A is true).
In contrast, a CASE expression forces the engine to evaluate every WHEN branch in order – often resulting in a table scan and no index usage【9†L54-L61】.

2️⃣ Translating real‑world filters

🔹 Example 1 – “Return all rows unless a specific employee is requested”

What we want: if @ReturnAll = 1 → return every row; otherwise → return rows where EmpID = @EmpID.

CASE version (hard to read)Boolean version (clean)
 
WHERE EmpID = CASE WHEN @ReturnAll <> 1
                    THEN @EmpID
                    ELSE EmpID END
 
WHERE (@ReturnAll = 1) OR (EmpID = @EmpID)
*If @ReturnAll = 1 the first predicate is true and the whole WHERE clause short‑circuits – the engine never touches EmpID.*

🔹 Example 2 – “Filter on a group of columns only when @Filter = 1”

Goal: when @Filter = 1 enforce Date = @Date, Cust = @Cust **and** Emp = @Emp. Otherwise ignore all three filters.

CASE version (three CASEs)Boolean version (single OR)
 
WHERE Date = CASE WHEN @Filter = 1 THEN @Date ELSE Date END
  AND Cust = CASE WHEN @Filter = 1 THEN @Cust ELSE Cust END
  AND Emp  = CASE WHEN @Filter = 1 THEN @Emp  ELSE Emp  END
 
WHERE @Filter <> 1
   OR (Date = @Date AND Cust = @Cust AND Emp = @Emp)
*Only when @Filter <> 1 does the optimizer need to evaluate the three column predicates – the rest of the rows are filtered instantly.*

🔹 Example 3 – “Optional single‑column filter (the classic ISNULL/COALESCE pattern)”

Typical code:

WHERE CustID = ISNULL(@CustID, CustID)   -- or COALESCE(@CustID, CustID)

Problems:

  • The expression defeats index usage (SQL Server must apply the function to every row)【11†L41-L63】.
  • Nullable columns cause three‑valued logic headaches (the predicate can become UNKNOWN).

Boolean rewrite (the gold standard):

WHERE (@CustID IS NULL) OR (CustID = @CustID)
*If the input parameter is NULL the first part evaluates to TRUE, short‑circuiting the index‑seek on CustID. This pattern works on every RDBMS and keeps the query SARGable (search‑argument‑able).*

3️⃣ When you really need CASE in a WHERE clause

There are a few edge‑cases where you must decide on a value before the predicate can be built (e.g. dynamically choosing a column). In those rare scenarios you can still avoid CASE by using dynamic SQL or splitting the logic into two separate statements (one for each branch). All other conditional logic belongs to the Boolean realm described above.

⚡ Performance snapshot – CASE vs Boolean

Gordon Linoff summed it up perfectly:

“Yes. There is a major, major difference between using a CASE expression and boolean expressions in the WHERE clause. The optimizer understands boolean expressions, but CASE must be evaluated in the order written, which forces a table scan and prevents index usage【9†L54-L61】.”

In practice you’ll see execution plans where the Boolean version uses an Index Seek (or a range scan) while the CASE version falls back to an Clustered/Index Scan. The impact multiplies dramatically on large tables.

4️⃣ Cheat‑sheet – quick reference for the most common patterns

Pattern descriptionSQL (Boolean) Notes
Simple “IF A THEN B” (NOT A) OR B Direct translation of implication.
“IF A THEN B ELSE C” ((NOT A) OR B) AND (A OR C) Two‑clause form – rarely needed, but useful for full ternary logic.
“Optional filter: only apply when @P is NOT NULL” (@P IS NULL) OR (Column = @P) SARGable – uses an index on Column.
“Apply filter when @Flag = 1” (@Flag <> 1) OR (Column = @Value) Same as the ReturnAll example.
“Combine several optional columns” (@P1 IS NULL OR Col1 = @P1) AND (@P2 IS NULL OR Col2 = @P2) AND ... Each predicate short‑circuits independently.

5️⃣ TL;DR – The “golden rule”

Never put CASE (or ISNULL/COALESCE) around column references in a WHERE clause. Convert every conditional requirement to a plain Boolean expression using AND / OR / NOT.

Doing so gives you:

  • Portable code (works on every major dialect).
  • Readability that any junior developer can grasp.
  • Speed – the optimizer can use indexes and short‑circuit logic.

Give it a try in your next stored procedure – you’ll notice the execution plans getting dramatically tighter.
